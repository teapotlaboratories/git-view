Product design spec · for Compose handoff

A mobile code editor that reads like a workspace.

GitView is a native Android git editor with a full-tool Claude agent working the same repo. This redesign gives it a product identity and rebuilds the four weak points: an illegible agent chat, a permission system shown as jargon, no fluid path between editing and chatting, and e-ink treated as a recolor instead of a layout. One component set, two first-class visual languages — Standard and Color E-Ink.

Direction: Modern editor IBM Plex Sans + JetBrains Mono Jetpack Compose · Material 3 Bigme B7 Pro · Kaleido 3 ◐ Chat prototype is live — try it

Touch targets: 48dp min Standard, 56dp E-Ink (fixes today's 34–36dp).

Shape · border · elevation

  • Standard: 12dp corners (cards/sheets), 8dp (chips/inputs), 999dp (send). Depth = 1dp tonal surface + soft shadow; borders only on inputs.
  • E-Ink: 6dp corners, no shadow ever — depth = a 1.5px border. Selected = filled surface-variant, not glow.
  • Diff gutter width 22dp both profiles; hairline dividers 1px / 1.5px (E-Ink).

Motion policy

  • Standard: M3 emphasized easing, 200–300ms; ripple on press; per-token chat streaming; skeleton shimmer.
  • E-Ink: none. No ripple, overscroll, or shimmer. Press = instant 1-bit invert. Streaming commits whole lines, not tokens.
  • Navigation & long lists paginate in discrete full-screen repaints (refresh-friendly) rather than scroll.

02

Editing and the agent, one workspace

Today the app is a flat five-screen stack with back-arrow the only way out, and no persistent path between Browse and Chat — you reach the agent through a small chip and return with Back. The two core loops (edit / drive-the-agent) are strangers. The fix collapses Browse + Chat into a single Workspace: a segmented switch on phone, a split pane on tablet. Diff and History become overlays over it, not separate destinations.

Permission model

The six raw profiles (read-only, confined, acceptEdits, auto, dontAsk, bypass) are a security control shown as jargon — auto, dontAsk and bypass carry wildly different risk with no cue. Renamed to plain language, ordered by escalating risk, each with an explicit allow/deny and a 0–4 risk signal. The power user keeps every level; the default is safe.

07

Implementation notes

What drops onto stock Material 3 versus what needs a custom composable, then a build order sequenced by impact — Chat and the tokens lead, exactly as expected.

Maps to stock Material 3

TopAppBar SegmentedButton ScrollableTabRow ListItem Elevated/OutlinedCard Filter/AssistChip DropdownMenu ModalBottomSheet AlertDialog OutlinedTextField Filled/IconButton LinearProgressIndicator Badge Snackbar

Needs a custom composable

ToolActivityCard — tool-typed header + expandable body (args / result / diff).
DiffRow / DiffView — gutter + syntax; shared by the Diff overlay and Chat.
RiskMeter — segmented bars + label, both profiles.
WorkspaceScaffold — adaptive segment (phone) / 2–3-pane split (tablet).
EinkPaginator — discrete full-page repaints replacing scroll.
ProfileTheme — token swap + motion-policy gate (disables ripple/anim on E-Ink).
StreamingText — per-token (Standard) / per-line (E-Ink).
SUGGESTED BUILD ORDER · HIGHEST IMPACT FIRST
1
Design tokens + ProfileTheme
Both ColorSchemes, Typography, spacing/shape objects, and the profile/motion gate. Everything else depends on it.
2
Chat transcript ★
LazyColumn + StreamingText + ToolActivityCard + DiffRow over the WebSocket. The priority — biggest legibility win.
3
Permission model
Tier list, ModalBottomSheet picker, RiskMeter, inline approval. Unblocks safe auto-edit and builds trust.
4
Workspace IA
SegmentedButton (phone) + split (tablet); Diff / History become overlays.
5
Connections & Repos status
Reachability polling, latency, git-state chips.
6
E-Ink profile pass
EinkPaginator, no-motion gate, weight/underline semantics, 56dp targets.
7
States polish
Loading skeletons, empty, error, offline across every screen.
08

Assumptions & open decisions

What I assumed to keep the spec buildable, and the calls that are yours to make before implementation.

ASSUMPTIONS
  • The bridge exposes /health for reachability + latency, and streams tokens and tool events over one WebSocket.
  • Per-turn / session cost + budget are reported by the bridge — the client only displays them.
  • Kaleido 3 color is muted and low-PPI, so a single accent is used sparingly; all real semantics ride on weight / underline / gutter.
  • Single user, personal use — no multi-account, no client-side git-graph rendering.
  • The Sora editing surface is fixed; only its surrounding chrome is designed.
DECISIONS · RESOLVED
✓
Default tier = Ask first for every new session.
✓
"Allow for session" permanently upgrades the tier (→ Auto-edit) until changed — as the prototype does.
✓
Tablet split divider is user-draggable (default ~55 : 45 editor : chat, position persisted).
✓
Remote / Local SDK provider is set per connection (on the bridge), not per session.
✓
Branch switching is in scope for v1 (the ref chip), alongside commit message + push.
✓
At 100% budget → warn (banner + red bar); never hard-stop mid-turn.
